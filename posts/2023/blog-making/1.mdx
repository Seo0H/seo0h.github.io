### 프레임워크 고르기: Next js vs Gasby

은근 고민하는데 시간이 많이 걸렸던 부분이에요. 개발 블로그를 직접 만든다 했을때, 여러가지 방법이 있겠지만 제가 편하다고 많이 들었던 기술은 [Gatby](https://www.gatsbyjs.com/) 였어요. 주변 분들도 해당 기술을 이용해 기술 블로그를 만들었더니 편하더라~ 는 이야기를 들었던 상태이기도 했구요.


이번 블로그를 만들기 전에 공부삼아 Gasby의 튜토리얼을 공부한 적이 있었는데, 해당 튜토리얼도 블로그 만들기 였을 만큼 관련 플러그인 생태계가 잘 구성 되어 있는 것 같더라구요.

#### Gatby의 장점, 특징

```
1. 플러그인 하나만 설치하면 번거로운 작업을 한번에 대체할 수 있다는 점
2. SSG (Static Site Generation) 최적화가 매우 잘 되어있다는 점
```


이전에 [Gatby 튜로리얼](https://www.gatsbyjs.com/docs/tutorial/)을 따라하며 놀랐던 점은, 위에서 말한 첫번째 장점처럼 **플러그인을 이용해 귀찮은 작업들을 한번에 해결**할 수 있다는 거였어요. (이미지 최적화, place-holder처리 등 ..)

추가로 블로그를 만들기 위해 Gatby를 크게 고려했던 이유는, 역시 플러그인 중 **mdx를 파싱해주는 플러그인**이 있다는 것이였어요. 이것 역시 정말 편하기도 하고 공식문서에서 튜토리얼로 해당 플러그인을 이용해 블로그를 만드는 과정을 알려주더라고요.

#### Gatby의 단점

```
1. 플러그인 끼리의 의존성 지옥
2. 러닝 커브 존재
```

많이 이것저것 저도 사전 조사를 하며, 많은 분들이 프로젝트의 사이즈가 커질수록 편리함을 주던 플러그인 때문에 나중엔 어려움에 빠지게 되는 경우가 있다는 것을 알게 되었어요.

일명 '의존성 지옥' 이라고들 많이 표현 하시던데, 알게모르게 하나의 플러그인이 의존하고 있는 플러그인이 많아 용량도 많이 잡아먹고, 버전업을 한번 하게 되면 모두 다 같이 버전업을 해야 하는 상황이 온다는 것을 알게 되었어요. 그러면서 따라오는 여러 사이드 이팩드들도 존재하고요.

추가로 당시 상황에서 저에게 포함되는 이야기지만, GraphQL을 새로 배워야 하고, Gasby에 익숙해지기 위해 공부를 새로 해야 하는 상황이였어요. 따라서 자연스럽게 Next js에 눈이 더 가게 되었답니다. ~~(Next js도 아직 잘 못다루기 때문이죠..)~~

제가 Gasby를 깊게 공부한 것이 아니고, 겉핧기 식으로 튜토리얼만 따라 해 본 입장이에요. 그래서 아마 제가 말하는 단점을 다 극복 할 방법이 있을 것 같아요.

---

#### Next js 의 장점, 특징

```
1. 유연한 의존성과 보다 자유로운 데이터 패칭
2. SSR(Server Side Rendering) 지원
2. 익숙함
```
리서치를 해 보았을 때 Next js는 상대적으로 Gasby보다는 데이터 패칭과 의존성에 있어 자유롭다는 평가가 많았어요.

SSR을 사용하기에 대량의 데이터들이 빠르게 변경되는 환경에 적합하다는 의견이 주를 이뤘답니다. 서버의 자원을 이용해 클라이언트의 부담을 줄이는 방식으로 운영이 되기 때문이죠.


#### Next js 의 단점, 그럼에도 선정한 이유

```
1. Gasby에 비해 사용 가능한 플러그인이 적다는 것
2. Gasby에 비해 속도가 약간 느리다는 것
```

사실 블로그만 만든다고 하면, 위의 장단점을 따져보았을때 - Gatby를 사용했을 것 같아요. 하지만 현재 취업 시장에서 요구하는 스택이 Next js가 압도적으로 많기도 하고, 이후 확장될 홈페이지에 대한 계획을 대략적으로 가지고 있기 때문에 최종적으로는 **Next js를 사용**하게 되었어요.

### 추가 사용 기술

추가로 코드 안정성을 위해 `TypeScript`를 사용하고, 반복되는 컴포넌트들이 있을 예정이기에 공통 CSS를 공유하며 쉽게 확장할 수 있는 CSS-in-JS 라이브러리인 `Styled-Components` 를 이용해 스타일링을 하기로 결정했어요.
